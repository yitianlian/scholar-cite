## What worked

- Scholar search via `scholarly` returned the correct paper.
- `cluster_id` extraction from `url_scholarbib` worked (`5Gohgn6QFikJ`).
- `/scholar?output=cite` HTML popup was fetched successfully on the first request.
- All 5 text formats (MLA / APA / Chicago / Harvard / Vancouver) were parsed correctly.
- Non-interactive ambiguity handling: both result candidates returned.

## What was blocked

- The second candidate (`cluster_id: 4ZRunyZN6eIJ`) returned **HTTP 403** on its cite
  popup — Scholar rate-limited us after the first successful fetch.
- The 4 machine-readable export formats (**BibTeX / EndNote / RefMan / RefWorks**) were
  not observed in this live run. The initial cite popup succeeded, but the subsequent
  export-link fetches were blocked. This is the first thing to verify once cookies are
  warmed up (via the Playwright captcha flow) or with the SerpAPI backend.

## How the export-format code was verified anyway

The parsing logic for all 9 formats is covered by unit tests against a saved HTML
fixture that mirrors Scholar's actual popup markup (`tests/fixtures/cite_popup_sample.html`).
`tests/test_citation.py::test_fetch_citation_set_composes_all_nine_formats` exercises
the full pipeline with a mocked fetcher and verifies that all 9 fields get populated.

```
pytest tests/ -v
============================== 7 passed in 0.07s ===============================
```

## Implications for the roadmap

- The captcha recovery flow described in `docs/design.md` §6 is **not optional** —
  Scholar's rate limiter is aggressive enough that even a small number of sequential
  cite/export fetches will be throttled. The Playwright-based cookie harvesting should
  be the next implementation priority.
- A SerpAPI fallback (design §5.4) will likely be the more reliable path for users who
  don't want to interact with a browser window.
- The 3-second inter-request delay in the design is probably too low for Scholar in 2026;
  we should bump to 5–8s + jitter for batch runs.
